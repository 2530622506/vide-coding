# GESP 题型分类与知识点对齐网站：结构化分析

## Executive Summary

这个产品不是做一个「搜索页」，而是做一个 GESP 题型分类与知识点对齐网站。核心任务是：持续搜集 GESP 相关题源，把不同网站上的题目统一归档，判断它属于几级、属于哪个算法范畴、这个算法范畴下有哪些题目，并说明每道题考了哪些知识点。

产品的第一性目标是「对齐」：

- 对齐官方大纲：题目是否属于对应等级。
- 对齐题源网站：同一道题在官方、OJ、题解站、训练题单里的不同版本。
- 对齐算法范畴：如数论、二分、链表、贪心、递归、分治、高精度、复杂度。
- 对齐题型模板：如二分答案判定型、质因数分解型、链表操作模拟型、排序贪心型。
- 对齐知识点：把每道题拆成可学习、可复习、可统计的知识点。

对 GESP 五级来说，动态规划不是核心分类。五级应优先围绕二分、链表、数论、递归、分治、贪心、高精度、复杂度等官方范围展开。简单动态规划属于六级，复杂动态规划属于七级；如果第三方题解把五级题标成 DP，系统应保存为「社区推断 / 冲突标签 / 待复核」，而不是直接当作官方分类。

结论：Go。范围判定为 large，因为它至少包含数据源注册、爬虫/解析、题目去重、等级对齐、算法范畴分类、知识点抽取、人工复核和分类目录前台。

## Product Shape

网站的主要页面应该是分类目录，而不是单一搜索框。

1. 等级目录
   - 一级到八级。
   - 每个等级展示官方知识点、题型分布、真题数量、样题数量、待复核题目数量。

2. 算法范畴目录
   - 数论、二分、链表、贪心、递归、分治、高精度、复杂度、排序/模拟等。
   - 每个算法范畴下列出覆盖等级、典型题型、代表题、知识点树。

3. 题型目录
   - 例如「二分答案判定型」「质因数分解型」「gcd/lcm 变形型」「链表插入删除模拟型」「排序关键字贪心型」。
   - 每个题型页列出：适用等级、前置知识、题目列表、常见坑点、证据来源。

4. 题目详情页
   - 标准题名、来源网站、原始链接、等级、语言、考试时间、题型、算法范畴、知识点、证据片段、置信度、人工复核状态。

5. 来源对齐页
   - 展示同一道题在官方 PDF、洛谷、Hydro、ACGO、博客题解等来源中的映射关系。
   - 标明哪个是官方来源，哪个是练习镜像，哪个只是题解或标签参考。

来源分层原则：

- 官方来源决定「几级」和「官方知识点范围」。
- OJ 镜像决定「哪里能练」和「题目 ID 映射」。
- 题解/博客决定「候选算法标签」和「解法证据」，但必须经过冲突检测。

## Classification Output Contract

每道题最终要输出一个统一分类对象：

```json
{
  "canonical_title": "奖品兑换",
  "exam": "GESP",
  "level": 5,
  "language": "C++",
  "session": "2025-06",
  "source_links": [
    {
      "type": "official_pdf",
      "url": "...",
      "trust": "canonical"
    },
    {
      "type": "oj_mirror",
      "url": "...",
      "trust": "practice"
    }
  ],
  "algorithm_domains": ["二分", "数学推导"],
  "problem_types": ["二分答案判定型"],
  "knowledge_points": ["单调性", "check 函数", "整数边界", "不等式变形"],
  "syllabus_fit": "exact",
  "confidence": 0.86,
  "review_status": "needs_review"
}
```

关键字段说明：

- `level`：优先由官方 PDF 标题、官方真题页、题目名中的等级解析得出。
- `algorithm_domains`：粗粒度算法范畴，用于目录聚合。
- `problem_types`：更贴近考试的题型模板，用于学习和复习。
- `knowledge_points`：最细粒度知识点，用于统计、推荐和讲解。
- `syllabus_fit`：`exact | adjacent | out_of_level | community_inferred | disputed | needs_review`。
- `review_status`：`extracted | inferred | confirmed | rejected | disputed`。

## Level 5 Taxonomy Seed

五级初始分类建议如下，后续由爬虫和人工复核持续扩展。

| 算法范畴 | 题型模板 | 知识点 | 候选代表题 |
|---|---|---|---|
| 数论 | 质因数分解型 | 质数、试除法、唯一分解定理、指数统计 | 小杨的锻炼、因数分解、B-smooth 数、奇妙数字 |
| 数论 | gcd/lcm 变形型 | 欧几里得算法、最大公约数、差分 gcd、最小公倍数 | 最大公因数、小杨的锻炼 |
| 数论 / 筛法 | 素数统计型 | 埃氏筛、线性筛、最小质因数、互质 | 数字选取 |
| 二分 | 二分答案判定型 | 单调性、check 函数、上下界、不等式 | 奖品兑换 |
| 链表 / 线性表 | 操作模拟型 | 插入、删除、遍历、节点关系维护 | 小杨的队列 |
| 贪心 / 排序 | 排序关键字贪心型 | 排序规则、局部最优、交换思想 | 成绩排序、挑战怪物、武器强化、数字选取 |
| 递归 / 分治 | 排序与过程分析型 | 递归边界、归并排序、快速排序、分治思想 | 待从选择题和代码阅读题中抽取 |
| 高精度 | 数组模拟运算型 | 进位、借位、按位存储、长整数运算 | 待从官方 PDF 中抽取 |
| 复杂度 | 复杂度判断型 | 多项式、对数复杂度、循环嵌套 | 选择题 / 判断题 |

注意：这张表是 seed，不是最终事实。比如「奖品兑换」在不同题解中可能被标成二分、数学推导甚至背包，系统要保存多来源证据，并通过官方大纲和人工复核给出最终分类。

## Data Acquisition Pipeline

1. Source discovery
   - 从官方真题页发现考试场次和 PDF。
   - 从官方样题页发现样题 PDF。
   - 从洛谷题单、zqiceberg、OJ 标签页发现镜像题目。

2. Raw document ingestion
   - 保存 URL、抓取时间、hash、来源类型、license/terms 状态。
   - PDF 原文、HTML 正文、题解正文分开存储。

3. Problem extraction
   - 官方 PDF：抽取考试时间、语言、等级、选择题、判断题、编程题、题号、题名。
   - OJ 题目页：抽取题号、题名、题面、标签、来源描述。
   - 题解页：抽取算法标签、关键代码信号、知识点描述。

4. Dedup and alignment
   - 用 `GESPYYYYMM level title`、题名相似度、题面 hash、OJ ID 建立候选映射。
   - 同一道题允许多个来源版本，但只生成一个 canonical problem。

5. Classification
   - 先用官方大纲做等级范围约束。
   - 再用规则识别算法范畴和题型模板。
   - 再用题解和代码信号补充候选知识点。
   - 最后进入人工复核队列。

6. Conflict handling
   - 如果官方等级和社区标签冲突，保留冲突并标记 `disputed`。
   - 如果第三方标注 DP 但官方等级为五级，默认不能直接归为五级 DP；需要解释为「社区解法标签 / 越级解法 / 待复核」。

## Classification Rules

建议从规则分类开始，不要一开始就完全依赖 LLM。

1. 等级识别
   - 从官方真题页、PDF 标题、题目标题中的 `GESP202506 五级` 提取。
   - 官方来源优先级高于 OJ 或博客。

2. 算法范畴识别
   - 数论：`gcd`、最大公因数、最小公倍数、质因数、素数、筛法、同余、原根。
   - 二分：二分答案、最大化最小值、最小化最大值、单调判定、`check(mid)`。
   - 链表 / 线性表：节点、前驱后继、插入、删除、队列维护。
   - 贪心：排序后选择、局部最优、最小代价、最大收益、交换论证。
   - 递归 / 分治：归并、快排、递归过程、分治合并。
   - 高精度：大整数、数组存位、进位、借位。
   - 复杂度：`O(log n)`、循环嵌套、对数复杂度、多项式复杂度。

3. 题型模板识别
   - 「能否达到某个值」且答案有单调性：二分答案判定型。
   - 「分解一个数 / 统计质因子指数」：质因数分解型。
   - 「多次查询 gcd」：gcd 变形型。
   - 「按多关键字排序并输出排名」：排序规则模拟型。
   - 「修改若干对象使目标最优」：贪心 / 枚举目标值型。
   - 「选择题问算法性质」：概念判断型。

4. 知识点抽取
   - 官方大纲词汇命中。
   - 题面关键词命中。
   - 题解步骤命中。
   - 代码结构命中。
   - 人工复核确认。

## MVP

第一个可交付版本建议做成「五级题型分类目录」，而不是搜索页：

1. 建立 source registry，接入官方大纲、官方真题页、官方样题页。
2. 抓取并解析 C++ 五级官方 PDF 与样题 PDF。
3. 接入洛谷题单和 zqiceberg 五级页，做题目 ID 和练习链接映射。
4. 生成五级题目总表：题名、年份、来源、等级、语言、题型、算法范畴、知识点、证据。
5. 生成五级算法范畴页：数论、二分、链表、贪心、递归/分治、高精度、复杂度。
6. 生成每个算法范畴下的题目列表和知识点覆盖表。
7. 做人工复核后台：确认分类、驳回分类、合并重复题、处理冲突标签。

MVP 不需要先做复杂搜索。分类目录、题目矩阵、知识点覆盖表更重要。

## Risks

版权风险最高。不要默认转载完整题面；先保存元数据、短证据片段、源链接和 hash。需要完整题面时，确认授权或只给用户本地导入能力。

来源质量不一致。官方是等级和真题事实来源；OJ 是练习入口；博客和题解只是标签线索。系统必须把这些来源分层展示。

分类冲突会经常出现。比如第三方题解可能把五级题标为 DP 或背包，但官方大纲并不把 DP 放在五级。系统不能简单覆盖，要保留证据、冲突和复核状态。

PDF 抽取质量不稳定。需要 parser version、原文坐标或页码、OCR fallback，并保留原始文档 hash 便于重跑。

## Recommended Roadmap

1. Source foundation：来源注册表、官方大纲、官方真题页、官方样题页。
2. Problem ingestion：PDF 解析、OJ 镜像解析、题目去重、canonical problem。
3. Classification engine：等级识别、算法范畴分类、题型模板分类、知识点抽取。
4. Catalog UI：等级目录、算法范畴页、题型页、题目详情页、来源对齐页。
5. Review workflow：人工确认、冲突处理、版本记录、分类导出。
6. Expansion：从五级扩到一至八级，从 C++ 扩到 Python，再加入题解质量评分和学习路径。

## Go / No-Go

Go，但必须坚持「来源分层 + 证据优先 + 人工复核」。如果只把网上题目抓下来堆成列表，价值不大；如果能把每道题对齐到等级、算法范畴、题型模板和知识点，它就会变成一个真正有用的 GESP 题型分类资料库。
