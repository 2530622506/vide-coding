# Frontend Guidelines

## 1. 组件选择优先级

前端开发时，组件与页面能力的选择顺序固定为：

1. `ProComponents`
2. `Ant Design`
3. 自定义组件

解释：

- 如果 `ProComponents` 已经能直接覆盖需求，优先使用。
- 如果 `ProComponents` 不合适，再使用 `Ant Design` 基础组件。
- 只有在前两者都不适合时，才允许写自定义 UI。

## 1.1 数据请求与状态管理优先级

前端请求与状态管理优先级固定为：

1. `ahooks` 的 `useRequest`
2. `ProComponents` 的 `request`
3. `Zustand`

解释：

- 页面和 feature 级数据请求优先使用 `useRequest`
- 如果组件本身天然适配 `ProComponents` 的 `request` 机制，则优先复用组件内建能力
- 多页面共享、跨组件同步、需要持久化的前端状态，优先考虑 `Zustand`

禁止：

- 随意手写重复的 `fetch + useEffect + useState` 模板代码
- 在没有明确理由时引入更重的状态管理方案
- 把请求状态、展示状态、全局状态混在一个大 hook 里

## 2. 设计原则

必须遵循 `KISS` 原则：

- 简单直接
- 避免过度包装
- 避免为了“看起来高级”而引入复杂交互
- 先保证可读、可维护、可验证

禁止：

- 过度设计页面层级
- 组件职责模糊
- 为了局部效果引入复杂抽象

## 3. 类型安全

前端必须启用并遵守 `TypeScript` 严格模式。

硬性要求：

- 禁止使用 `any`
- 优先使用明确的接口和类型别名
- 外部接口返回值必须声明类型
- 组件 `props` 必须显式声明类型
- 状态对象必须显式建模，不允许“随手塞字段”

## 4. 质量红线

以下规则属于硬性约束：

- 单文件代码行数 `< 1000`
- 圈复杂度 `< 15`
- 注释密度 `> 10%`

解释：

- 文件过长必须拆分
- 分支过多必须拆解为更小的函数或组件
- 对复杂逻辑、状态流、边界条件必须写清楚注释

## 5. 目录组织

前端目录按职责拆分，不允许把页面、请求、状态、展示混写。

推荐结构：

```text
src/
  app/
  layouts/
  pages/
  features/
  components/
  hooks/
  services/
  types/
  utils/
```

约束：

- `pages` 只负责页面组装
- `features` 承载领域能力
- `services` 只处理请求与数据访问
- `types` 统一放类型定义
- `components` 放可复用展示组件

## 6. 组件职责

### Page

- 负责布局和拼装
- 不承载大段业务逻辑

### Feature Component

- 负责领域交互
- 允许少量本地状态

### Shared Component

- 不感知具体业务上下文
- 可被多处复用

## 7. 状态管理

默认优先级：

1. `ahooks useRequest`
2. `feature` 级 hook
3. `Zustand`
4. 浏览器持久化（如 `localStorage`）

说明：

- 远程数据请求优先交给 `useRequest`
- 全局共享状态优先交给 `Zustand`
- `localStorage` 只用于轻量持久化，不替代状态管理

## 8. 注释要求

注释不是装饰，必须服务于维护性。

必须写注释的场景：

- 状态切换逻辑不直观
- 请求回退策略
- 复杂条件判断
- 组件之间的数据依赖
- 与后端接口约定有关的特殊处理

不需要写注释的场景：

- 变量名和函数名已经足够表达语义
- 只是重复代码表面行为

## 9. 页面体验原则

- 先保证数据真实，再谈动效
- 优先补齐 loading / empty / error / success 状态
- 结果页优先支持查看详情
- 问答页优先支持引用追溯
- Markdown 报告展示先遵循 KISS，当前阶段允许直接展示文本
- 后续如果展示复杂度上升，再引入成熟插件做增强

## 10. 执行要求

后续所有前端新增代码都必须默认对照本规范自检：

- 是否优先用了 `ProComponents / Ant Design`
- 是否保持 `KISS`
- 是否避免了 `any`
- 是否违反文件长度或复杂度红线
- 是否为复杂逻辑补了注释
